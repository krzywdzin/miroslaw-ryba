# Phase 5: Environment Setup - Research

**Researched:** 2026-03-21
**Domain:** Pipeline stage 2 -- entity review, agent profile display, per-simulation parameter override, simulation launch
**Confidence:** HIGH

## Summary

Phase 5 builds a review/confirmation page between graph construction (Phase 4) and simulation execution (Phase 6). Users inspect what the system extracted (entities, agent profiles) and adjust simulation parameters before committing to a run. The page uses a tab-based layout with three tabs: Encje (entities table), Agenci (agent profile cards), and Parametry (parameter override form). A persistent "Uruchom symulacje" CTA with confirmation dialog transitions to Phase 6.

The entire API surface already exists in `src/api/simulation.ts` with typed methods: `getEntities`, `getProfiles`, `create`, `prepare`, `prepareStatus`, and `start`. Zod schemas exist for all response shapes. The feature follows established project patterns: Zustand persist store for pipeline state, TanStack Query for data fetching, React Hook Form + Zod for parameter forms, i18n namespaces for translations. The entity detail view reuses the Sheet (slide-out panel) pattern from `NodeDetailPanel.tsx`.

**Primary recommendation:** Create `src/features/environment/` following the exact patterns from `src/features/graph/` -- Zustand store for environment pipeline step, TanStack Query hooks for API calls, tab-based page component composing entity table, agent cards grid, and parameter override form. Install `@radix-ui/react-alert-dialog` for the launch confirmation dialog.

### Pattern 2: Entity Data Fetching (follow useGraphData)
**What:** TanStack Query hook wrapping simulationApi.getEntities.
**When:** Entities tab loads.

```typescript
export function useEntities(graphId: string | null) {
  return useQuery({
    queryKey: ['simulation', 'entities', graphId] as const,
    queryFn: () => simulationApi.getEntities(graphId!),
    enabled: !!graphId,
    staleTime: Infinity, // entities don't change after graph construction
    select: (response) => ({
      entities: response.data?.entities ?? [],
      entityTypes: response.data?.entity_types ?? [],
      count: response.data?.filtered_count ?? 0,
    }),
  })
}
```

### Pattern 3: Per-Simulation Parameter Override
**What:** Local state form that reads config store defaults but does NOT write back to config store. Values are only used when calling simulation create/start.
**When:** Parametry tab.

```typescript
// Read defaults from config store, but form state is local
```

### Pattern 4: Prepare Status Polling (follow useBuildStatus)
**What:** Use usePollingQuery to poll simulation prepare status.
**When:** After user launches and backend starts preparing environment.

```typescript
export function usePrepareStatus(taskId: string | null) {
  return usePollingQuery(
    ['simulation', 'prepare', 'status', taskId] as const,
    () => simulationApi.prepareStatus({ task_id: taskId! }),
    {
      enabled: !!taskId,
      isComplete: (data) => {
        const status = data.data?.status
        return status === 'completed' || status === 'failed'
      },
      interval: 2000,
    },
  )
}
```

### Anti-Patterns to Avoid
- **Writing parameter overrides to config store:** Per-simulation params are ephemeral. Saving them pollutes the global config with per-run values.
- **Fetching entities without graphId:** The entities endpoint requires graphId from the graph store. Guard with `enabled: !!graphId`.
- **Building custom sort/filter for table:** Use @tanstack/react-table or simple array.sort -- don't build a generic table framework.
- **Navigating to environment page without completed graph:** Must check graphStore.step === 'done' and graphStore.graphId is set.

## Common Pitfalls

### Pitfall 1: Stale graphId/projectId
**What goes wrong:** Environment page renders but graphId is null because user navigated directly.
**Why it happens:** Zustand persist restores state, but if user cleared storage or opened new tab, IDs are missing.
**How to avoid:** Route guard checks graphStore.step === 'done' && graphStore.graphId !== null. Redirect to /graph if not.
**Warning signs:** TanStack Query hooks disabled (enabled: false), empty page renders.

### Pitfall 2: Simulation Create vs Prepare Flow
**What goes wrong:** Calling prepare before create, or calling create twice.
**Why it happens:** Two-step API: first `simulationApi.create()` returns simulation_id, then `simulationApi.prepare()` starts agent generation.
**How to avoid:** Clear sequence: create -> store simulationId -> prepare -> store taskId -> poll prepareStatus.
**Warning signs:** 400 errors from backend, missing simulation_id in prepare call.

### Pitfall 3: Entity Relationships Not in Entity List Response
**What goes wrong:** Entity table shows entities but clicking shows no relationships.
**Why it happens:** The entities endpoint returns `{ uuid, name, type, description }` without relationship data. Relationships come from graph data (edges).
**How to avoid:** For entity detail, fetch graph data (already cached from Phase 4 with staleTime: Infinity) and filter edges by entity uuid. OR use `simulationApi.getEntity(graphId, uuid)` for single entity details.
**Warning signs:** Empty relationship list in detail panel.

### Pitfall 4: Auto-save Confusion
**What goes wrong:** Parameter changes on environment page unexpectedly update global settings.
**Why it happens:** Copying SimulationSection.tsx pattern which auto-saves to config store via debounce.
**How to avoid:** Environment parameter form must NOT auto-save to config store. Form state is local. Values are read at launch time only.
**Warning signs:** Global settings page shows values changed after environment page visit.

### Pitfall 5: Missing AlertDialog Component
**What goes wrong:** Import error for AlertDialog component.
**Why it happens:** Not yet installed. Only Dialog-based Sheet exists.
**How to avoid:** Install @radix-ui/react-alert-dialog and create src/components/ui/alert-dialog.tsx following shadcn pattern (manual creation per project convention).

### Data Flow for This Phase
```
1. Page loads:
   - Read graphId from useGraphStore
   - Fetch entities via simulationApi.getEntities(graphId)
   - Read config defaults from useConfigStore

2. User reviews entities (Tab 1) and agents (Tab 2)
   - NOTE: Agent profiles may not exist yet until prepare is called
   - Initially show empty state or placeholder on Agenci tab

3. User adjusts parameters (Tab 3)
   - Local form state, NOT persisted to config store

4. User clicks "Uruchom symulacje":
   a. simulationApi.create({ project_id, graph_id, enable_twitter, enable_reddit })
   b. Store simulation_id in environment store
   c. simulationApi.prepare({ simulation_id, entity_types, use_llm_for_profiles: true })
   d. Store task_id, start polling prepareStatus
   e. When prepare completes, fetch profiles
   f. Show confirmation dialog with final summary
   g. simulationApi.start({ simulation_id, max_rounds })
   h. Navigate to /simulation (Phase 6)
```

**Important clarification on flow:** The exact ordering of prepare vs. review depends on the backend. Looking at the architecture doc, the flow is:
1. User lands on environment page AFTER graph is ready
2. Entities are already available (extracted during graph construction)
3. User reviews entities, adjusts params, then clicks launch
4. Launch triggers: create simulation -> prepare (generates agents) -> poll until ready -> start simulation
5. Agent profiles appear DURING/AFTER prepare, not before

This means the Agenci tab may initially show "will be generated when simulation launches" or the prepare step could be triggered earlier to pre-generate profiles for review. The user decision says "User can view generated character profiles" suggesting profiles exist before launch. This means prepare should run BEFORE the launch confirmation, likely triggered when user first enters the environment page or via a "Generate Agents" action.

**Recommended flow:**
1. Page loads -> auto-create simulation + auto-prepare (generates agents)
2. Poll prepare status -> when done, fetch and display profiles
3. User reviews all three tabs
4. User clicks "Uruchom symulacje" -> confirmation -> start

## Open Questions

1. **When to trigger simulation prepare?**
   - What we know: Profiles need to exist for user to review them (ENVR-02, ENVR-04)
   - What's unclear: Should prepare auto-run on page load, or require explicit user action?
   - Recommendation: Auto-trigger create+prepare on page mount. Show loading state on Agenci tab while preparing. This gives users entities to review immediately while agents generate in background.

2. **Entity relationship data source**
   - What we know: Entity schema has uuid, name, type, description. No relationship field.
   - What's unclear: Does `getEntity(graphId, uuid)` return relationships? Or must we use graph edges from Phase 4?
   - Recommendation: Reuse cached graph data from useGraphData (staleTime: Infinity) and filter edges by entity ID, same pattern as NodeDetailPanel.

3. **Profile schema completeness**
   - What we know: ProfileSchema has agent_id, name, personality, stance, platform. All optional except agent_id and name.
   - What's unclear: Where is "memory state" and "opinion timeline" data? (Required by ENVR-04)
   - Recommendation: Start with available fields. Memory/stance may be in the `prepare_info` unknown field of PrepareStatus, or may need schema extension. Flag as LOW confidence.
