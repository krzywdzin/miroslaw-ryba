    **docs/installation.md:**

    Create the docs/ directory if it does not exist, then create docs/installation.md with this structure:

    1. **Title:** "# Installation Guide"

    2. **Prerequisites section:**
       - Docker >= 24.0 and Docker Compose v2 (for backend)
       - Node.js >= 18.0 (for frontend)
       - npm >= 9.0
       - Git

    3. **Option 1: Docker (Recommended)** -- step by step:
       - Step 1: Clone the repository: `git clone https://github.com/OWNER/miroslaw-ryba.git && cd miroslaw-ryba`
       - Step 2: Configure environment: `cp .env.example .env` then edit with API keys (link to configuration.md)
       - Step 3: Start MiroFish backend: `docker compose up -d`
       - Step 4: Verify backend is running: `docker compose ps` should show mirofish-backend healthy; `curl http://localhost:5050/health` should return OK
       - Step 5: Install frontend dependencies: `npm install`
       - Step 6: Start frontend dev server: `npm run dev`
       - Step 7: Open http://localhost:5173 in browser

    4. **Option 2: Source Deployment** -- for development without Docker:
       - Backend: Requires Python 3.10+, clone MiroFish repo, install Python deps, run `python backend/run.py`
       - Frontend: Same as Docker steps 5-7
       - Note: Docker is strongly recommended as the backend has complex Python dependencies

    5. **Production Build section:**
       - `npm run build` creates optimized build in dist/
       - `npm run preview` to test production build locally
       - Serve dist/ with any static file server

    6. **Troubleshooting section:**
       - Port conflicts: change BACKEND_PORT and VITE_PORT in .env
       - Backend not starting: check `docker compose logs mirofish-backend`
       - Health check failing: backend takes ~30s to start (start_period in compose)

    7. **Link to Configuration Guide** at bottom: "Next: [Configuration Guide](configuration.md)"

    **docs/configuration.md:**

    Create docs/configuration.md with this structure:

    1. **Title:** "# Configuration Guide"

       Optional variables table:

       | LLM_BOOST_API_KEY | Secondary LLM key for faster processing | (empty) |
       | LLM_BOOST_BASE_URL | Secondary LLM endpoint | (empty) |
       | LLM_BOOST_MODEL_NAME | Secondary model name | (empty) |
       | BACKEND_PORT | Backend port mapping | 5050 |
       | VITE_PORT | Frontend dev server port | 5173 |

    4. **LLM Configuration section:**
       - Supported providers: any OpenAI-compatible API (Alibaba DashScope, OpenAI, Azure OpenAI, local Ollama)
       - Recommended model: qwen-plus (default, good balance of cost and quality)

       - Setup: create account at zep.ai, get API key from dashboard
       - Test: use "Test Connection" in Settings panel

    6. **Docker Management section:**
       - The Settings panel shows Docker container status
       - Start/stop/restart containers from the UI
       - View container logs for debugging

    7. **UI Configuration section:**

       - Settings persist in localStorage across browser sessions

       1. Fork and clone the repository
       2. Install dependencies: `npm install`
       3. Start backend: `docker compose up -d`
       4. Start dev server: `npm run dev`
       5. Run tests: `npm run test`
       6. Link to docs/installation.md for detailed setup

    3. **Available Scripts section:**
       | Command | Description |
       |---------|-------------|
       | `npm run dev` | Start development server |
       | `npm run build` | Build for production |
       | `npm run test` | Run tests (Vitest) |
       | `npm run test:watch` | Run tests in watch mode |
       | `npm run lint` | Run ESLint |
       | `npm run format` | Format code with Prettier |
       | `npm run typecheck` | Run TypeScript type checking |

    4. **Code Style section:**
       - TypeScript with strict mode
       - ESLint 9 flat config (eslint.config.mjs) with typescript-eslint and react-hooks plugins
       - Prettier: semi: false, singleQuote: true, trailingComma: all, printWidth: 100, tabWidth: 2
       - Run `npm run lint` and `npm run format` before committing

    5. **Project Structure section:**
       ```
       src/
       ├── api/          # API client modules (typed with Zod)
       ├── app/          # App shell, router, providers
       ├── components/   # Shared UI components (shadcn/ui)
       ├── features/     # Feature modules (one per pipeline stage)
       │   ├── chat/
       │   ├── dashboard/
       │   ├── environment/
       │   ├── graph/
       │   ├── reports/
       │   ├── settings/
       │   └── simulation/
       ├── hooks/        # Shared React hooks
       ├── i18n/         # i18n configuration
       ├── lib/          # Utility functions
       ├── locales/      # Translation files (pl/, en/)
       ├── stores/       # Zustand state stores
       ├── styles/       # Global styles, Tailwind CSS v4
       └── types/        # Shared TypeScript types
       ```

    6. **Internationalization section:**
       - All UI strings must go through react-i18next
       - Add translations to both src/locales/pl/*.json (Polish - default) and src/locales/en/*.json (English)
       - Polish has 4 plural forms (CLDR categories) -- use ICU MessageFormat for plurals
       - *Uwaga: Polski jest jezykiem domyslnym aplikacji. Wszystkie nowe teksaty musza miec tlumaczenie polskie.*

    7. **Pull Request Process section:**
       1. Create a feature branch from main
       2. Make changes with tests where applicable
       3. Ensure `npm run lint && npm run test && npm run typecheck` all pass
       4. Submit PR with clear description of changes
       5. PRs should be focused -- one feature or fix per PR

    8. **Issue Reporting section:**
       - Use GitHub Issues for bug reports and feature requests
       - Include: steps to reproduce, expected behavior, actual behavior, browser/OS info
       - Label bugs with "bug", features with "enhancement"
